# IWE Web IM 前端产品需求文档 (PRD)

## 1. 文档信息
- **版本**: V1.0
- **日期**: 2026-05-12
- **描述**: 基于 IWE 后端 API 的轻量化、多账号管理 Web IM 前端系统。

## 2. 项目背景与目标
旨在为 IWE 用户提供一个高性能、轻量化的 Web 端工作台。系统核心目标是：
- **多账号支持**: 能够在一个界面内同时挂载并管理多个微信账号。
- **极致轻量**: 消息不入库（仅存内存），每会话修剪至 50 条，降低浏览器负载。
- **消息安全与完整**: 解决 WebSocket 与 HTTP 同步间的时序竞争，确保不丢消息、不重消息。
- **撤回拦截**: 满足极客需求，支持对撤回消息的拦截展示。

## 3. 功能需求规格 (Functional Requirements)

### 3.1 引导与配置
- **服务器配置**: 首页支持输入 API Base URL 及 Admin Token。
- **验证机制**: 提交前验证 API 连通性。
- **持久化**: 配置信息保存于 LocalStorage，刷新不丢失。

### 3.2 多账号管理
- **账号多开**: 左侧第一栏显示已登录账号列表。
- **添加账号**: 点击 `+` 呼出扫码弹窗，调用 `/login/GetLoginQrCode`。
- **账号切换**: 点击头像切换当前视图，各账号数据完全隔离。
- **状态感知**: 非活动账号收到新消息时，头像显示红点提醒。

### 3.3 消息同步系统 (核心)
- **初始化同步**: 登录后拉取 `/message/GetContact` 获取最近会话。
- **Redis 快照**: 调用 `/other/GetRedisSyncMsg` 快速填充最近消息。
- **增量补录**: 建立 WS 后调用 `/message/SyncMsg(0)`。
- **去重逻辑**: 维护 5 分钟滑动窗口，基于 `NewMsgId` 进行全局去重。

### 3.4 聊天功能
- **消息气泡**: 支持文本、图片（占位展示）、系统消息。
- **撤回拦截**: 收到 10002 类型消息时，不删除原气泡，更新 UI 标记为“已撤回（拦截）”。
- **原图获取**: 消息流默认加载缩略图，点击后按需请求高清图。
- **发送逻辑**: 支持乐观更新（即时渲染气泡 + 转圈）。

## 4. 技术架构与规范 (Technical Design)

### 4.1 四栏布局设计
- **第一栏 (Account Bar, 60px)**: 账号切换、加号按钮。
- **第二栏 (Nav Bar, 60px)**: 消息列表切换、通讯录切换。
- **第三栏 (Chat List, 280px)**: 最近会话、最后消息预览、未读计数。
- **第四栏 (Chat Window, Auto)**: 聊天标题、消息流、输入区。

### 4.2 消息归一化解析器 (MessageParser)
所有消息必须经过以下转换逻辑：
- 识别 `MsgType`。
- 对 XML 格式的内容进行正则表达式或 DOM 解析。
- 提取关键字段：`from`, `to`, `content`, `type`, `isSelf`, `isRevoked`。
- 将归一化后的 JSON 对象存入 Pinia。

### 4.3 存储策略
- **内存存储 (Pinia)**: 
  - `chats: Record<accountUuid, Record<wxid, AppMessage[]>>`
- **修剪算法**: 
  - 每次 `push` 消息后，检查对应 `wxid` 数组长度，若 `> 50` 则 `shift()`。

### 4.4 同步时序保障方案
1. 开启 WebSocket 监听并进入“缓冲模式”。
2. 发起 HTTP `SyncMsg` 请求。
3. 待 HTTP 返回后，先处理 HTTP 消息，再处理缓冲队列中的消息。
4. 解除缓冲模式，进入实时模式。

## 5. 非功能性需求
- **响应式**: 适配主流 PC 浏览器。
- **容错性**: WebSocket 断线后自动指数级退避重连。
- **安全性**: 敏感 Session Key 仅在请求头/URL 中传递，不暴露在 UI 界面。

## 6. 开发计划
- **Phase 1**: 基础框架与多账号 Store 搭建。
- **Phase 2**: 解析引擎与全量同步逻辑开发。
- **Phase 3**: 四栏 UI 开发与 WebSocket 联调。
- **Phase 4**: 优化撤回拦截与图片按需加载体验。
